### 5.1 vLLM 기동 옵션 ([`run_vllm_model_server.sh`](artifacts/scripts/run_vllm_model_server.sh))
```bash
vllm serve "${MODEL_PATH}" \
  --download-dir "${MODEL_CACHE_DIR}" \
  --served-model-name "${SERVED_MODEL_NAME}" \
  --tokenizer unsloth/gpt-oss-20b-BF16 \
  --trust-remote-code \
  --dtype bfloat16 \
  --host 127.0.0.1 --port 8000 \
  --tensor-parallel-size 1 \
```

- `MAX_MODEL_LEN=131072` — 모델 네이티브 context를 그대로 노출
- `TP_SIZE=1` — 단일 GPU로 BF16 21B 적재 가능 (≈ 130 GB VRAM/GPU)

## 6. 트러블슈팅 및 해결

| # | 증상 | 원인 | 해결 |
|---|------|------|------|
| 1 | `--orchestrate` 모드 첫 실행이 `FileNotFoundError: '/usr/bin/torchrun'`로 즉시 실패 | 시스템 `python` 사용 시 `sys.executable` 기반 torchrun 탐색이 system path를 가리킴 | 모든 호출이 학습 venv `python` 절대경로를 사용하도록 daemon 및 docs 정정 |
| 2 | vLLM 스트리밍 응답의 `delta.content`가 빈 문자열로만 흐름 | vLLM 0.13이 `model_type=="gpt_oss"`에 harmony 파서를 강제 적용 → 파인튜닝 모델 평문 출력을 analysis 채널로 오인식 | `VLLM_DISABLE_HARMONY=1` env var로 우회. `serving_chat.py` 패치로 env 인식 |
| 3 | OpenWebUI에서 MCP 도구 호출이 무한 retry 루프 | gpt-oss-20b-128k 모델이 `tool.query_etf_data` 형태로 호출, OpenWebUI dispatcher는 정확 매칭만 수행 | `middleware.py` dispatcher 두 곳에 prefix tolerant lookup 폴백 추가 |
| 4 | OpenWebUI 사용 중 `max_tokens 3072 > 4096 - 1197` 에러로 응답 차단 | vLLM이 `--max-model-len=4096`으로 기동되어 컨텍스트 협소 | `MAX_MODEL_LEN=131072`로 확장 (모델 native limit 그대로) |
| 5 | daemon이 메모리에 캐싱된 옛 스크립트로 실행 | bash 인터프리터가 스크립트를 메모리에 로드한 후 파일 수정 미반영 | daemon 재기동 절차를 표준 운영 절차로 명문화 |
| 6 | 검증 응답 시 ngrok abuse warning HTML이 반환되어 OpenWebUI 응답 멈춤 | ngrok 무료 도메인이 브라우저 User-Agent에 한해 경고 HTML 반환 | 1회 "Visit Site" 클릭으로 쿠키 설정 → 이후 정상 통신 |
